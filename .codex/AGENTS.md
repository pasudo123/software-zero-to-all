# Global AGENTS.md

## 기본 원칙

- 프로젝트별 `AGENTS.md` 또는 `agents.md`가 있으면 해당 파일을 먼저 따른다.
- 전역 지침과 프로젝트 지침이 충돌하면 더 가까운 위치의 구체적인 프로젝트 지침을 우선한다.
- 관련 없는 수정은 피하고, 요청 범위에 맞는 작고 집중된 변경만 반영한다.
- 사용자가 만든 변경은 되돌리지 않는다. 의도하지 않은 변경이 섞여 있으면 범위를 확인하고 필요한 파일만 수정한다.

## 작업 정책

- 작업 전후에는 가능하면 `git status -sb`로 현재 브랜치와 변경 상태를 확인한다.
- 변경 파일이 없으면 커밋하지 않는다.
- 기능, 문서 정리, 설정 변경처럼 성격이 다른 변경은 가능한 한 분리한다.
- 명시적 요청 없이 코드나 문서의 구조를 크게 재작성하지 않는다.

## Git/PR 정책

- 가능하면 작업 브랜치에서 변경하고, `main` 직접 작업은 지양한다.
- PR 대상 브랜치는 기본적으로 `main`으로 한다.
- PR 제목과 본문은 기본적으로 한글로 작성한다. type, 브랜치명, 명령어, 파일명, 기술 용어는 영어를 그대로 사용해도 된다.
- PR 하나에는 하나의 목적만 담고, 변경 배경, 주요 변경 사항, 검증 방법/결과, 리스크 또는 롤백 방법을 포함한다.
- 커밋을 추가해 PR의 변경 의도나 검토 포인트가 달라졌다면 PR 제목과 본문도 최신 상태로 갱신한다.

## 도구 사용

- 검색은 `rg` 또는 `ripgrep`를 우선 사용한다.
- 파일 탐색은 `fd` 또는 `rg --files`를 우선 사용한다.
- JSON 확인은 `jq`를 우선 사용한다.
- GitHub 연동 작업은 `gh`를 우선 사용한다.

## 문서 작성 기준

- 문서와 설명은 한글을 기본으로 하고, 기술 용어가 더 자연스러운 경우 영어를 함께 쓴다.
- AI가 쓴 티가 강한 문장, 과한 요약, 홍보 문구, 근거 없는 단정은 줄인다.
- 기존 문서의 문체와 구조를 가능한 한 유지한다.
- Markdown 링크는 가능하면 상대 경로를 사용한다.

## 보안

- 실제 API key, access token, refresh token, token cache, `.env`, 비밀번호, 개인 계정 정보, 내부 URL은 커밋하지 않는다.
- 문서 예시는 `<API_KEY>`, `<DB_PASSWORD>`, `<HOST>` 같은 placeholder를 사용한다.
- GitHub Actions secret 이름은 문서화할 수 있지만 실제 값은 저장소에 남기지 않는다.

## 검증

- 변경 범위에 맞는 테스트, 빌드, 수동 확인을 수행한다.
- 문서만 변경한 경우에도 가능하면 `git diff --check`를 확인한다.
- 실행하지 못한 검증이 있으면 이유와 남은 리스크를 명확히 남긴다.
