# AGENTS.md

## 목적

이 문서는 `software-zero-to-all` 저장소에서 문서를 추가하거나 정리할 때 따르는 프로젝트별 작업 가이드이자 Git/PR 워크플로우의 단일 진실 공급원(SSoT)이다.

이 저장소는 애플리케이션 코드가 아니라 소프트웨어 학습, 실무 메모, 명령어 사용법, 인프라/DB/백엔드 개념을 누적하는 Markdown 기반 지식 저장소다. 정책 변경이 필요하면 이 파일을 먼저 수정하고, 다른 문서에는 동일한 규칙을 중복 작성하지 않는다.

## 구조

루트 `README.md`는 전체 문서의 인덱스 역할을 한다. 새 문서를 추가했을 때 여러 번 다시 볼 만한 내용이면 `README.md`에도 링크를 추가한다.

주제별 문서는 가장 가까운 디렉터리에 둔다.

- `database/`: MySQL, 트랜잭션, 인덱스, 락, 파티션 등 DB 정리
- `command/`: curl, jq, awk, grep, lsof 같은 CLI 사용법
- `infra/`: AWS, Docker, Kubernetes, Kafka 등 인프라 메모
- `dummy/`: 아직 정식 분류 전이거나 단편적인 실무/학습 메모
- `book/`, `inflearn/`, `udemy/`: 책과 강의 기반 정리
- `personal/`: 개인 관점의 설계, 개발 습관, 개념 정리
- `piece/`: 짧은 단편 지식과 작은 운영 메모
- `image/`: 문서에서 참조하는 이미지와 다이어그램

## 컨벤션

문서는 짧은 개요에서 시작하고, 이후 원인, 동작 방식, 예시, 해결 방법, 참고 순서로 정리한다. 기존 문서처럼 `## 개요`, `## 원인`, `## 해소방법`, `## reference` 같은 실용적인 제목을 우선 사용한다.

코드 예시는 fenced code block으로 작성하고 가능한 언어를 명시한다.

```bash
curl -v https://example.com
```

```kotlin
CoroutineScope(Dispatchers.IO + SupervisorJob())
```

Mermaid 다이어그램은 필요한 경우에만 사용하고, 복잡한 설명을 보완하는 용도로 둔다. 링크는 가능하면 상대 경로를 사용한다. 파일명은 기존 관례처럼 한글과 영어를 허용하되, 검색 가능한 핵심 키워드를 포함한다.

새로 작성하는 학습/실무 메모는 일자별로 정리한다. 날짜가 중요한 글이나 연속 학습 문서는 `yyyy-MM-dd_topic-kebab.md` 형식을 우선 사용한다. 예: `2026-04-30_redis-lock.md`. `README.md`, `AGENTS.md`처럼 표준 파일명이 있는 경우만 예외로 둔다.

디렉터리명은 소문자 kebab-case를 사용한다. 대문자, 공백, 언더바를 섞지 않는다. 예: `object-oriented-programming/`, `operating-system/`, `book/http-완벽가이드/`. 단, Java 리소스 표준인 `META-INF`처럼 외부 규약이 강한 이름은 유지한다.

새 글이나 기존 글 수정은 AI 초안/검수 흐름을 기본으로 한다. 작성자는 초안의 의도와 맥락을 제공하고, AI는 기술적 정확성, 링크, 날짜, 파일명, README 인덱스 반영 여부를 함께 검토한다. 틀린 표현은 바로잡되, 개인 학습 메모의 말투와 뉘앙스는 가능한 유지한다.

## 내용 및 PR

새 글은 하나의 주제를 중심으로 작성한다. 문제 상황, 핵심 개념, 실제 사용 예시, 주의사항을 함께 남기면 재사용성이 높다. 명령어 문서는 옵션별 예시를 포함하고, Spring/Kotlin/DB 문서는 동작 조건과 실패 케이스를 함께 적는다.

문서를 추가하거나 이름을 바꾼 뒤에는 `README.md`의 항목별 인덱스를 함께 갱신한다. 같은 항목 안에서는 작성일 순으로 배치하고, 분류가 애매한 글은 임시로 `dummy/`에 두되 이후 주제가 명확해지면 가장 가까운 디렉터리로 이동한다.

모든 변경은 작업 브랜치와 `main` 대상 PR을 통해 반영한다.

1. 로컬 `main`을 최신 상태로 갱신한다.
2. 새 작업 브랜치를 생성한다. 예: `docs/<topic>`
3. 문서를 수정하고 필요한 확인을 수행한다.
4. 작업 브랜치에서만 커밋하고 원격에 푸시한다.
5. `main` 대상 PR을 생성한다.
6. PR 머지 후 로컬/원격 작업 브랜치를 정리하고 `main`으로 복귀한다.

가능하면 `git worktree`를 사용해 작업 브랜치별 독립 작업 공간에서 수정한다. 기본 작업 폴더는 가능한 `main` 상태로 유지하고, worktree를 사용할 수 없는 경우 PR 본문에 사유를 남긴다.

커밋 메시지는 최근 히스토리와 맞춰 짧게 작성한다.

- `docs : probe`
- `docs : redis setnx + expire + uuid`
- `docs 정리 : 짧게 기억 정리.`

PR 제목과 본문은 기본적으로 한글로 작성한다. PR 하나에는 하나의 목적만 담고, 본문에는 변경 배경, 주요 변경 사항, 확인한 내용, 리스크나 참고사항을 간단히 적는다. 긴 PR 본문은 `gh pr create --body-file` 사용을 우선한다.

CLI 도구는 목적에 맞는 전용 도구를 우선 사용한다. 검색과 파일 탐색은 `ripgrep(rg)`와 `fd`, JSON 확인은 `jq`, HTTP/API 확인은 `curl` 또는 `httpie`, 반복 관찰은 `watch`, GitHub 연동은 `gh`를 기본 도구로 사용한다. 명령어를 문서에 남길 때는 실행 목적, 주요 옵션, 대표 예시를 함께 적는다.

PR 전에는 `git diff --check`를 실행하고, Markdown 내부 링크와 이미지 경로가 실제 파일을 가리키는지 확인한다. 대규모 경로 변경 시에는 이전 디렉터리명이나 파일명이 남아 있지 않은지 `rg`로 점검한다.

실제 토큰, 비밀번호, 개인 서버 주소, 내부 URL은 문서에 남기지 않고 `<HOST>`, `<DB_PASSWORD>` 같은 placeholder를 사용한다.
